# 保險條款自動化採集與文本轉換系統

## 🛠 專案架構與分支功能說明

專案內部分為三個核心模組，各自承載專案在不同開發階段的任務：

### 1. `Scratcher/`（自動化網頁爬蟲模組）

負責對保發中心老舊且非標準架構的網頁進行動態操作與 PDF 批次下載。內含依開發日期重構之演進版本：

* **`20260604/`（初始版本）：** 第一代基礎管線，直接對全文檢索頁面（`QueryFullText.aspx`）進行暴力檢索，在處理動態渲染與網頁防禦時易引發超時崩潰。
* **`20260605/`（DOM 狀態偵測）：** 診斷出目標網址錯誤。修正為針對險種分類商品頁（`Query.aspx`）進行爬取，排除框架切換邏輯錯誤。
* **`20260619/`（分頁與點擊優化）：** 修正網頁內 `__doPostBack` 之非同步點擊機制，處理動態換頁與保單細節頁（`Item.aspx`）的下載連結點擊。
* **`20260620/`（穩定優化版）：** * `v4.1`：引入動態等待，解決下載視窗未渲染即關閉的問題。
* `v4.2`：增加檔案下載防掉落檢驗機制與多線程監控。
* `v4.3`：修復在下載大量保單時引發的非預期中斷與網絡重連機制。



### 2. `Convert to TXT/`（文本轉換模組）

負責將下載完成的法律/醫學格式保單條款 PDF 轉為標準純文字。

* **核心功能：** 透過 `docling` 與 `pypdf` 套件，批次讀取 `INPUT_DIR` 的 PDF 檔案，將格式複雜的表格、除外責任與條文轉換為無損的 `.txt` 檔案，輸出至 `OUTPUT_DIR`。

### 3. `Selection and Convert/`（已停用/歷史模組）

* **分支說明：** 原計畫用於依據使用者理賠常見問題之關鍵字（如：門診手術、特定傷病）盲測 PDF 前兩頁以進行保單篩選。
* **狀態更新：** **專題團隊於 6/20 晚間決議，現階段暫不做 Selection 篩選流程。** 主因為保單條款多採用嚴謹法律字眼，以特定病名（如智齒、粉瘤）盲測易失準。故此模組目前僅作留存，系統流程改為**爬蟲後直接進行全量轉檔**。

## 開發紀錄 (STAR 框架說明)

### 📌 Situation (情境)

在開發「智能保險檢索與比較系統」時，需要大量且完整的保險條款作為知識庫。然而，目標數據源（保發中心商品查詢系統）架構老舊，具備非標準的 ASP.NET 表單防禦、非同步 `__doPostBack` 換頁機制，且保單條款皆為非結構化的 PDF 格式，無法直接投入後續數據應用。

### 📌 Task (任務)

1. **自動化採集：** 建立穩定之網頁爬蟲管線，批次下載全量健康保險條款 PDF。
2. **架構調整：** 依據 6/20 決策，精簡處理管線，取消關鍵字篩選（Selection）階段，改為全量處理。
3. **格式轉換：** 將採集到的 PDF 條款檔案，轉換為機器可讀的 TXT 格式。

### 📌 Action (行動)

1. **網頁逆向與重構（`Scratcher/`）：** * 捨棄不穩定的全文檢索爆破，改定位險種分類頁（`Query.aspx`）。
* 針對 `v4.1 - v4.3` 進行連續迭代，利用 `WebDriverWait` 監聽真實下載狀態，克服網頁非同步關閉視窗造成的數據丟失。


2. **管線精簡調整：** 於 6/20 晚間停止 `Selection and Convert` 中的篩選機制，避免機械化關鍵字比對造成核心條文遭到截斷。
3. **文本結構化轉換（`Convert to TXT/`）：** 撰寫 `convert_pdfs.py`，使用 `docling` 解析引擎批次處理高密度文本與表格，排除掃描雜訊，輸出至標準文字檔。

### 📌 Result (結果)

1. **管線穩定度提升：** 爬蟲腳本經 `v4.3` 修正後，成功克服老舊網頁動態加載限制，可無間斷進行大批量保單採集。
2. **數據流暢度優化：** 移除篩選機制後，直連 PDF 轉 TXT 轉換管線。轉換後的純文字檔案完整保留了「門診手術」、「除外責任」等核心理賠判定條文，數據結構完整，可直接對接後續 RAG 知識庫之切片（Chunking）與向量化需求。
